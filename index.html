// 01 — Workloads

What is a
Workload?

A workload is a pre-packaged bundle of tools, SDKs, libraries, and templates installed into Visual Studio to support a specific development scenario. For mobile, the key workload is "Mobile development with .NET".

📦

Android SDK & Emulator

Full Android toolchain including build tools, platform SDK, and AVD emulator for testing on virtual devices.

🍎

iOS Build Tools

Xamarin.iOS / MAUI iOS support. Requires a connected Mac or Mac build host to compile iOS apps.

🧩

.NET MAUI Templates

Project templates, item templates, and scaffolding for cross-platform mobile and desktop apps.

📡

NuGet & Dependencies

Package manager integration with thousands of mobile-ready libraries pre-configured for cross-platform use.

🖥️

Device Simulators

Android Virtual Devices (AVD) and iOS Simulator support for running apps without a physical device.

🔧

Hot Reload & Debug

XAML Hot Reload and .NET Hot Reload allow live UI changes without restarting the app during development.

// 03 — Forms & MAUI

What are
Forms & MAUI?

Xamarin.Forms (older) and .NET MAUI (modern) are cross-platform UI frameworks. You write one UI in XAML and MAUI converts it into native controls on each platform.

OLDER

Xamarin.Forms

Separate Android/iOS projects. Still supported but in maintenance mode. Being replaced by MAUI.

MODERN ✓

.NET MAUI

Single project targeting Android, iOS, macOS & Windows. Evolved successor to Xamarin.Forms.

How XAML becomes native UI XAML → Native
// You write once in XAML: <Button Text="Click Me" BackgroundColor="Blue" /> // .NET MAUI converts it to: // Android → android.widget.Button // iOS → UIButton (UIKit) // Windows → WinUI Button // macOS → NSButton (AppKit) // One codebase → Real native controls on every platform
// 04 — Languages

Languages
& Their Roles

Mobile development with Visual Studio primarily uses C# and XAML, with supporting languages for configuration and data handling.

C#
All app logic, models, services, ViewModels, commands, data binding, API calls. The primary language — everything runs through it.
XAML
UI layout and design. Like HTML for apps — defines pages, buttons, labels, layouts, styles, and data binding expressions.
XML
Platform configuration files: AndroidManifest.xml (permissions), Info.plist (iOS settings), NuGet package configs.
JSON
API response data, appsettings.json for environment config, launchSettings.json for debug profiles.
SQL
Local SQLite database queries for offline storage, caching, and persisting user data on the device.
// 05 — Classes & Types

Key Classes
& Types

Each class serves a specific architectural role. Click a tab to see the code and explanation.

App.xaml.cs — Application entry point C#
public partial class App : Application { public App() { InitializeComponent(); // Set the first screen of your app MainPage = new AppShell(); } protected override void OnStart() { // Called when app first launches } protected override void OnSleep() { // Called when app goes to background } }

The Application class is the root of your app. It controls global resources, startup logic, and lifecycle events (start, sleep, resume). Every app has exactly one.

MainPage.xaml.cs — A screen/page C#
public partial class MainPage : ContentPage { private readonly MainViewModel _viewModel; public MainPage(MainViewModel viewModel) { InitializeComponent(); _viewModel = viewModel; BindingContext = _viewModel; // Connect VM to UI } protected override async void OnAppearing() { base.OnAppearing(); await _viewModel.LoadDataAsync(); } }

ContentPage represents a single screen in your app. The BindingContext links the page to its ViewModel, enabling automatic data binding between UI and logic.

MainViewModel.cs — Business logic & state C#
public class MainViewModel : INotifyPropertyChanged { private string _title = "Loading..."; public string Title { get => _title; set { _title = value; OnPropertyChanged(); } } private ObservableCollection<Product> _products = new(); public ObservableCollection<Product> Products { get => _products; set { _products = value; OnPropertyChanged(); } } public ICommand LoadCommand { get; } public MainViewModel(ApiService apiService) { LoadCommand = new Command(async () => await LoadDataAsync()); } public async Task LoadDataAsync() { Title = "Products"; var data = await _apiService.GetProductsAsync(); Products = new ObservableCollection<Product>(data); } }

ViewModel holds all state and logic for a screen. INotifyPropertyChanged tells the UI to update when data changes. ICommand exposes actions that XAML buttons can trigger.

Product.cs — Data model C#
public class Product { public int Id { get; set; } public string Name { get; set; } public decimal Price { get; set; } public string ImageUrl { get; set; } public bool InStock { get; set; } public DateTime CreatedAt { get; set; } } // Used with SQLite for local storage: [Table("products")] public class ProductEntity { [PrimaryKey, AutoIncrement] public int Id { get; set; } public string Name { get; set; } }

Models are pure data containers — no logic, just properties. They represent data from your API or database and are passed between layers of the app.

ApiService.cs — External data fetching C#
public class ApiService { private readonly HttpClient _http; public ApiService(HttpClient http) { _http = http; _http.BaseAddress = new Uri("https://api.example.com/"); } public async Task<List<Product>> GetProductsAsync() { var json = await _http.GetStringAsync("products"); return JsonSerializer.Deserialize<List<Product>>(json); } public async Task<Product> GetProductAsync(int id) { var json = await _http.GetStringAsync($"products/{id}"); return JsonSerializer.Deserialize<Product>(json); } }

Services handle external communication — REST APIs, SQLite databases, authentication, file I/O. They're injected into ViewModels via dependency injection.

MauiProgram.cs — App bootstrap & DI C#
public static class MauiProgram { public static MauiApp CreateMauiApp() { var builder = MauiApp.CreateBuilder(); builder .UseMauiApp<App>() .ConfigureFonts(fonts => { fonts.AddFont("OpenSans.ttf", "OpenSansRegular"); }); // Register services (Dependency Injection) builder.Services.AddSingleton<ApiService>(); builder.Services.AddSingleton<DatabaseService>(); // Register pages & ViewModels builder.Services.AddTransient<MainPage>(); builder.Services.AddTransient<MainViewModel>(); return builder.Build(); } }

MauiProgram is the bootstrap file — runs before anything else. It configures the dependency injection container, registers services, fonts, handlers, and builds the app.

// 06 — Architecture

MVVM
Pattern

Model-View-ViewModel is the standard architecture for .NET MAUI apps. It separates concerns, making code testable, maintainable, and scalable.

VIEW

XAML pages and controls. Shows data to the user, captures input. Has no business logic.


MainPage.xaml
⟵ Data Binding ⟶

VIEWMODEL

Holds state & logic. Exposes Properties and Commands. Notifies View of changes via INotifyPropertyChanged.


MainViewModel.cs
⟵ Calls / Returns ⟶

MODEL

Pure data. No logic. Represents API responses, database records, or domain entities.


Product.cs
Layer Knows About Does NOT Know About Example Files

ViewModel Model, Services View (no UI references) *ViewModel.cs
Model Nothing else View, ViewModel, Services *.cs (in Models/)
Service Model, HTTP, DB View, ViewModel *Service.cs
// 07 — Execution Flow

How It All
Works Together

From app launch to a user seeing data on screen — here's the complete execution flow in a .NET MAUI application.

1. App Launches on Device

The OS starts the app. On Android this calls MainActivity.cs, on iOS it calls AppDelegate.cs. Both immediately hand control to .NET MAUI.

2. MauiProgram.cs Runs

The DI container is built. Services like ApiService and DatabaseService are registered. Fonts and handlers are configured. The app is constructed.

3. App.xaml.cs Sets MainPage

App.cs runs and sets MainPage = new AppShell(), which defines the navigation structure and first visible screen.

4. AppShell Defines Navigation

Routes are registered: //MainPage, //DetailPage etc. Shell handles tab bars, flyout menus, and URL-style navigation.

5. MainPage XAML Renders

XAML is parsed and native controls are created on the platform. Data bindings are established between the View and its ViewModel via BindingContext.

6. ViewModel Loads Data

OnAppearing() fires, triggering LoadDataAsync(). The ViewModel calls ApiService, which makes an HTTP request and returns typed Model objects.

7. UI Auto-Updates via Binding

When ViewModel sets Products = new ObservableCollection(...), INotifyPropertyChanged fires, and the bound ListView/CollectionView automatically refreshes — no manual UI update code needed.

8. User Interacts → Command Fires

A button tap triggers an ICommand in the ViewModel. The command runs business logic, updates state, and the cycle repeats — View always reflects the latest ViewModel state.
